# Technical Application Architecture

## 🏗️ Architecture Overview

The application is built using a modern **Next.js 15 (App Router)** architecture. It leverages server-side rendering (SSR) for SEO and performance, and client-side components for interactivity.

-   **Frontend**: React, Tailwind CSS 4, Radix UI.
-   **Backend**: Next.js API Routes (`app/api`), Server Actions.
-   **Database & Auth**: Supabase ("Backend as a Service").
-   **AI Processing**: Fal.ai (external GPU cluster).
-   **Storage**: Cloudflare R2.
-   **Payments**: Dodo Payments.

---

## 📂 System Taxonomy & Directory Layout

### `app/`
This is the core of the Next.js App Router.
-   `page.tsx`: The main landing page.
-   `dashboard/`: The private user area where the actual image restoration tools live.
-   `api/`: Backend API endpoints.
    -   `api/auth/`: Handles Supabase authentication callbacks.
    -   `api/webhooks/`: Listeners for external events (e.g., Dodo payment success).
-   `auth/`: Login and Signup pages.
-   `globals.css`: Global styles and Tailwind directives.

### `components/`
Reusable UI elements.
-   `ui/`: Base components (Buttons, Inputs, Modals) - likely generated by `shadcn/ui`.
-   `landing/`: Components specific to the marketing pages (Hero, Features, Testimonials).
-   `dashboard-client.tsx`: The main interactive component for the logged-in dashboard.
-   **Feature Components**: e.g., `nostalgic-hug-client.tsx`, `family-portrait-client.tsx` contain the specific logic for those tools.

### `lib/`
Utility functions and shared logic.
-   `api-client.ts`: Helper functions to make API requests to the backend.
-   `r2.ts`: Configuration and helpers for interacting with Cloudflare R2 storage.
-   `wordpress.ts`: Fetches blog posts for the blog section.

### `supabase/`
Contains database migrations and type definitions.
-   `migrations/`: SQL files that define the database schema.

## 🔑 Core Logic Flows

### 1. Authentication
Authentication is managed by Supabase.
-   **Flow**: User signs up -> Supabase sends confirmation email -> User clicks link -> Redirects to `app/auth/callback` -> Session established.
-   **Middleware**: Protected routes (like `/dashboard`) are guarded by `middleware.ts` which checks for an active Supabase session.

### 2. Image Restoration Process
1.  **Upload**: User drops an image in `image-upload.tsx`.
2.  **Storage**: The image is uploaded to Cloudflare R2 via `lib/r2.ts`.
3.  **Processing**: The frontend calls an API route (e.g., `app/api/restore/route.ts`).
4.  **AI Generation**: The API route sends the image URL to **Fal.ai**.
5.  **Result**: Fal.ai returns the restored image.
6.  **Persistence**: The result URL is saved to the Supabase database (linked to the user).

### 3. Commerce & Entitlement Engine (Credits)
-   Users purchase credits via **Dodo Payments**.
-   When a payment succeeds, Dodo sends a webhook to `app/api/webhooks/dodo`.
-   The webhook verifies the signature and updates the user's `credits` balance in Supabase.

### 4. User Feedback System
-   **Trigger**: The `useFeedback` hook tracks user actions. After a successful restoration and the first image download, it checks `/api/feedback` to see if the user should be prompted.
-   **UI**: The `components/feedback-modal.tsx` prompts the user for a rating (1-5) and optional text.
-   **Backend**: `app/api/feedback/route.ts` handles the logic (storing feedback, tracking "skipped" counts, etc.).

## 💾 Data Persistence Schema

The application likely uses the following core tables in Supabase:

-   **`profiles` (or `users`)**: Extends the default Supabase auth user. Stores credits, full name, avatar.
-   **`restorations`**: Tracks every image generation.
    -   `id`, `user_id`, `original_image_url`, `restored_image_url`, `status`, `created_at`.
-   **`transactions`**: Logs payment history.
    -   `id`, `user_id`, `amount`, `credits_added`, `provider_id` (Dodo transaction ID).

## 🎨 Styling

Styling is handled by **Tailwind CSS**.
-   **Customization**: Check `tailwind.config.ts` (or `globals.css` for v4 variables) to modify colors, fonts, and border radius.
-   **Dark Mode**: Implemented via `next-themes`.

## 🔄 Adding a New Feature

1.  **Backend**: Create a new API route in `app/api/new-feature/route.ts` to handle the logic.
2.  **Frontend**: Create a new client component in `components/new-feature-client.tsx`.
3.  **UI**: Add a new card or link in `app/dashboard/page.tsx` pointing to your new feature.
